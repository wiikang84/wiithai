# wiiInfo 점주(사장님) 기능 기획안

> 작성 2026-06-18 · 제품기획·UX팀 + 개발·아키텍처팀 합동 회의 종합
> 상태: **기획안 (구현 전)** — 형님 검토용. 급조 금지, 단계적 진행.

---

## 1. 배경 & 목표
- wiiInfo는 외국인이 자국어로 동네 로컬 가게를 찾는 발견 앱. 현재 점포 정보는 **운영자가 전부 입력**, 점주는 "우리 가게 등록 신청" 버튼(애널리틱스만)만 있음.
- **목표**: 점주가 직접 **영업 상태(특히 긴급 휴무)·영업시간·(음식점)메뉴**를 관리해, 외국인 사용자에게 **정확한 실시간 정보**를 제공. 동시에 점주를 앱에 묶어(리텐션) 해자(점주 관계)를 강화.

## 2. 핵심 원칙 (두 팀 합의 — 이게 기획의 뼈대)
1. **"점주 셀프 입력"에 기대지 않는다.** 동네 사장님(50~60대) 디지털 부담이 현실. 배민도 영업사원이 대행. → **운영자(영업) 대행 등록 + 점주는 긴급휴무 토글 하나**로 시작해 효용 증명.
2. **점주 화면은 100% 한국어.** 점주(한국인)와 사용자(외국인)의 언어를 분리. 점주는 다국어를 신경 쓸 필요 0.
3. **업종 차등.** 마트=영업시간+긴급휴무로 충분 / 음식점=메뉴+**식이태그(할랄·돼지고기없음)**. 마트에 메뉴판 UI 주면 이탈.
4. **기존 자산 보존.** 정적 점포 데이터(7개어 번역)는 그대로 두고, 점주가 바꾸는 **동적 데이터만 Firestore 오버레이**. 전면 이전 금지.
5. **안전 우선 다국어.** 할랄·알레르기 정보 오역 = 안전사고. 구조화 데이터(시간·가격)는 자동, 안전 텍스트는 운영자 검수 필수.

## 3. 업종별 점주 기능
**공통 (마트·음식점·할랄 전부)**
| 항목 | 비고 |
|---|---|
| 영업시간(요일별 open/close, 브레이크타임, 정기휴무) | **구조화 필수** — 자동판정 근거 |
| 긴급 휴무 토글("오늘 임시휴무") | 익일 0시 자동 해제 ← 안 끄는 사고 방지 |
| "지금 영업중" 자동 표시 | 점주 입력 아님, 시스템 산출 |
| 전화번호 | 이미 있음 |
| 대표 사진 1~3장 | 신규 |
| 한 줄 소개 | ko 입력 → 번역 |

**음식점 전용 (추가)**
| 항목 | 비고 |
|---|---|
| 메뉴(명+가격+사진) | 서브컬렉션 |
| 특별/추천 메뉴 슬롯 1~3 | 강조 |
| **식이 태그(할랄·채식·돼지고기없음·땅콩)** | ★ 외국인엔 메뉴 사진보다 중요한 차별점 |
| 포장/배달 토글 | 단순 |

## 4. 점주 UX 흐름
```
[내 정보 탭] → "사장님 모드"(기존 ownerRegisterButton 재활용)
  → Google 로그인(기존 재사용)
  → 내 가게 연결(운영자가 현장 영업 때 등록) 또는 신규 신청
  → [내 가게 관리]
       ┌ 큰 토글: [● 지금 영업중 / 오늘 임시휴무]  ← 최상단·최대 (킬러 기능)
       ├ 영업시간 편집 (요일 7칸 그리드, 시간은 드롭다운 = 타이핑 금지)
       │    └ "평일 동일" 복사 버튼, 브레이크타임은 접힘
       ├ 전화 / 사진 / 한 줄 소개
       └ (음식점만) 메뉴 관리
```
- **긴급 휴무 토글이 진짜 킬러 기능** — 점주가 매일 여는 이유. 갑자기 문 닫을 때 버튼 하나 → 사용자 상세에 즉시 "오늘 임시휴무" 배지.
- 영업시간은 **자유 텍스트 금지**(파싱·다국어·자동판정 불가). 드롭다운만.

## 5. 데이터 모델 — 하이브리드(정적 시드 + Firestore 오버레이)
**전면 Firestore 이전 반대.** 정적 7개어 번역은 거의 안 변하고, 매 로드 20개 문서 읽기는 비용·지연만 늘어남. 점주가 만지는 동적 필드만 Firestore.

```
data/places.js (정적, 유지)  →  앱 로드 시 PLACES 배열 (단일 진실원: id·이름·주소·번역·category·좌표)
        +
Firestore  storeStatus/{placeId}  ←  점주 동적 데이터만
        ↓
   렌더 직전 1곳에서 placeId로 머지 (Firestore 실패 시 정적 폴백 → 기존 동작 100% 보존)
```
**`storeStatus/{placeId}`** (점주 1명 = 1문서, placeId = 문서ID):
```
ownerUid, hours{mon:[{open,close}],…,sun:[]},
holidayClosed{active, until(KST), reason?},
manualOpenOverride: null|"open"|"closed",
updatedAt, createdAt
```
- **메뉴는 서브컬렉션** `storeStatus/{placeId}/menu/{itemId}` (음식점만, 상세 열 때 지연 로드)
- **기존 `hours`(자유텍스트)는 `legacyHoursText`로 보존** → 구조화 입력 없는 가게 폴백 (삭제 말고 보존 규칙 준수)
- 업종 차등은 **정적 `category`로 분기** (storeStatus에 중복 저장 안 함 = 정합성)

## 6. 권한 / 보안
- **MVP: 문서 `ownerUid` 방식** — 운영자가 현장 영업 때 점주 uid 수동 기입(현장 등록 모델과 일치). Custom Claims/Cloud Function은 점주 수백 명 넘으면 승격.
- **firestore.rules (신규 storeStatus)**:
  - `read: if true` / `update: if isOwner(placeId)` / **`create: if false`**(점포 생성=운영자만 → 가짜 점포 스팸 차단) / `delete: if false`
  - `ownerUid` **불변 보장**(소유권 탈취 방지) + 필드 화이트리스트·타입·길이 검증(기존 `validUserProfile` 패턴 복제)
- 사용자/점주 분리: 같은 Google 로그인. ownerUid 매칭 없으면 쓰기 자체가 거부 → 별도 계정 불필요.

## 7. "지금 영업중" 자동 판정 (★ 타임존 지뢰 주의)
- **반드시 KST 고정**(사용자 기기 로컬타임 의존 금지 — 과거 날짜 버그 이력 있음):
  ```js
  const kstNow = new Date(new Date().toLocaleString("en-US", { timeZone: "Asia/Seoul" }));
  ```
- 판정 우선순위: ① `manualOpenOverride==="closed"` → 닫힘 → ② 긴급휴무 → ③ `hours[요일]` 슬롯 매칭 → 영업중 → ④ 구조화 hours 없으면 "영업시간 확인 필요"(legacy 폴백)
- 클라이언트 계산 → **읽기 비용 0, 서버 함수 불필요**

## 8. 다국어 정책 (필드별 차등)
| 필드 | 처리 | 이유 |
|---|---|---|
| 영업시간·긴급휴무·전화·가격 | **자동 즉시 반영** | 숫자/구조화 = 번역 불필요 |
| 메뉴명·식이태그 | **기계번역 + 운영자 검수** | 할랄·알레르기 오역 = 안전사고 |
| 한 줄 소개 | 기계번역 자동 | 틀려도 위험 낮음 |
- 번역 없는 언어는 **ko 원문 + "번역 준비중" 폴백** (빈 언어 깨짐 방지)

## 9. 단계별 로드맵
| Phase | 범위 | 비고 |
|---|---|---|
| **1 (MVP·사전 영업용)** | `storeStatus` 도입. 점주: 영업시간(요일별)+긴급휴무/강제닫기 토글. 사용자: "영업중/종료/긴급휴무" 배지 + 구조화 영업시간. **메뉴·사진·쿠폰 제외** | 운영자가 storeStatus 문서 수동 생성(ownerUid 기입) |
| **2** | 음식점 메뉴 서브컬렉션(명+가격), 품절 토글, 식이태그 | 검수 큐 |
| **3** | 사진(Storage), 쿠폰 동적화, 점주 셀프 가입 + Custom Claims | 규모 확대기 |

## 10. 비용 (Spark 무료: 읽기 5만/쓰기 2만 일)
- 메인 로드당 `storeStatus` **1쿼리 일괄**(~20 doc read). DAU 수백까진 무료 내
- **SW에 5~10분 TTL 캐시**로 읽기 절감 (단 Firestore 응답 잘못 캐시 시 "영업중" 멈춤 주의 → 짧은 TTL)
- 점주 쓰기 하루 수십 건 = 무풍지대

## 11. 악용 방지 (MVP 최소)
1. 전화 인증(가게 등록 전화로 코드) — 가장 현실적
2. 운영자 1차 승인(신규 점주 연결 자동 활성화 금지)
3. 변경 이력 로깅(롤백·분쟁 대비)
4. 민감 필드 동결(가게명·주소·좌표·할랄인증은 점주 수정 불가, 운영자 데이터 보호)
5. 사용자 신고 버튼
- ⚠️ 과인증(사업자등록증·OTP앱)은 점주 이탈만 키움 → 전화+승인 2단이면 초기 충분

## 12. 리스크 체크
- **회귀**: 머지는 렌더 직전 1곳만, Firestore 실패/오프라인 시 정적 폴백(앱 안 죽게)
- **오프라인/PWA**: storeStatus는 네트워크 의존 → "영업중 배지 미표시 + 영업시간만"으로 graceful degrade. Firestore 호출은 SW 캐시 제외/짧은 TTL
- **타임존**: KST 미고정이 최대 지뢰. 기기 타임존 바꿔가며 테스트
- **소유권 탈취**: `create:false` + `ownerUid` 불변 검사 필수
- **자정 넘김 영업(18:00~02:00)**: MVP 제외 또는 `close<open`이면 익일 처리 한 줄

## ✅ 형님 결정 (2026-06-18 확정)
1. **MVP = 데모 수준 유지.** 점주 기능은 현재 정도(스텁)로 두고 풀 구현은 추후. 급조 금지.
2. **점주 등록 = master 계정의 등록폼**으로 점주·매장 등록. 점주 화면은 **`owner.html` 별도 페이지 분리**, **한 점주가 여러 매장 관리 가능**(배민식 멀티매장 → `where ownerUid==내uid`로 내 가게 목록 조회).
3. **착수 = 선행작업(다크모드 정리·메타 교체·실데이터) 이후.** 기획만 확정, 구현은 선행작업 뒤.
4. **워크플로우**: commit/push는 자율(매 작업), 배포는 형님 명시 지시 시만.

> 따라서 지금은 이 기획안을 확정해두고 **선행작업부터 진행**한다. 점주 기능 구현은 선행작업 완료 후 Phase 1(영업시간+긴급휴무, master 등록폼, owner.html 멀티매장)로 착수.

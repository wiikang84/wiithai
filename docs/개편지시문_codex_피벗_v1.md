## 0. 확정된 제품 정의 (모든 작업의 기준)

> **"한국 사는 외국인이 자기 나라 언어로, 우리 동네 로컬 음식점·마트를 찾는 발견 앱"**

- **주기능 = 정보공유 (로컬 발견)**: 외국 음식점·마트뿐 아니라 한국 로컬 맛집·가게까지. 핵심 질문은 "이 동네에 뭐가 있나".
- **배우기 = 서브 (유입용 후크)**: 사람을 끌어들이는 미끼 기능. 메인 아님. 기능은 유지하되 위계상 보조 탭.
- **언어 1회 선택 = 앱 전체의 단일 기준**: 접속 시 나라/언어만 고르면 그 뒤 **모든 화면·라벨·데이터가 그 언어로** 출력된다. 한국어 하드코딩 없음.

### 설계 1원칙 (가장 중요)
> **언어는 앱에 단 하나의 전역 상태다. 사용자가 한 번 선택하면, 그 이후 모든 출력(UI 라벨·정보 데이터·학습 데이터·안내 문구)이 예외 없이 그 언어를 따른다.**

4팀이 만장일치로 지목한 "정체성 분열(학습앱 머리 + 식료품점 몸통, 첫 화면 한국어)"은 이 1원칙을 끝까지 적용하면 대부분 해소된다.

## 2. P0 — 보안 하드닝 (먼저, 방향과 무관)

근거: 보안팀 진단.

1. **CSP에서 `'unsafe-inline'`(script-src) 제거** — `firebase.json` headers의 CSP(진단 시 L16, L37 부근). 코드에 인라인 `<script>`가 없으므로 제거해도 동작에 영향 없고 XSS 방어선이 올라간다. 제거 후 라이브에서 콘솔 CSP 위반 에러가 없는지 반드시 확인.
2. **`frame-ancestors 'none'`(또는 `'self'`) 추가** — 클릭재킹 차단. 현재 CSP에 없음.
3. **보안 헤더 보강** (`firebase.json`):
   - `Strict-Transport-Security: max-age=31536000; includeSubDomains`
   - `Permissions-Policy: geolocation=(self), camera=(), microphone=()`
4. **URL 출력 스킴 화이트리스트 헬퍼 도입** — `href`/`src`에 들어가는 값이 `http(s):`/`tel:`/`mailto:`/상대경로만 허용되도록 검증 함수 추가. 특히 `app.js`의 tel 분기(진단 시 L1327 `tel:${action.value}`)는 현재 escape가 누락돼 있다. **점포·점주 데이터를 Firestore에서 받기 시작하면 즉시 실위험으로 격상되므로, 외부 데이터 도입 전에 선제 적용**할 것.

> ⚠️ **콘솔 측 작업(코드 아님, 형님/관리자 처리 항목 — 지시문에만 기록)**: Google Cloud Console에서 Firebase apiKey에 HTTP referrer 제한(`https://wiiinfo.web.app/*`), Kakao Developers에서 JS 키 플랫폼 도메인 제한. 안 되어 있으면 무료 쿼터 도용 위험. codex 작업 아님.

## 3. P1 — 언어 전역화 + 진입 온보딩

근거: 디자인·개발·마케팅 공통 지목.

### 3-1. 진입 = 언어 선택 온보딩
- 현재 언어 선택이 "배우기" 탭 안에 묶여 있다(`learnView`). 이를 **앱 최초 진입 화면(온보딩)으로 승격**한다.
- 최초 실행(저장된 언어 없음) 시: 나라/언어 선택 화면을 먼저 보여주고, 선택 즉시 전역 언어 상태에 저장(localStorage) → 이후 진입부터는 바로 메인(발견)으로.
- `navigator.language`로 기본값을 추정해 제시하되, 사용자가 바꿀 수 있게 한다.
- 언어 변경 진입점은 앱 어디서나 접근 가능한 상단/설정에 상시 노출(1회 선택 후에도 변경 가능).

### 3-2. 전역 언어 상태 단일화
- 현재 정보(`PLACE_COPY`)와 학습 데이터가 **서로 다른 언어 집합**으로 따로 논다. 학습은 7개(ko/en/th/vi/ja/zh/es), `PLACE_COPY`(places.js)는 4개(ko/en/th/vi)뿐.
- **하나의 언어 상태(예: `state.lang`)를 단일 소스로** 두고, 모든 렌더 함수가 이 값을 참조하게 통일한다.
- 정적 UI 라벨 전체를 이 언어 상태에 연동: 탭/내비 라벨, 헤더, 버튼, 안내 문구.

### 3-3. 한국어 하드코딩 제거
- `index.html`의 `nearbyView`(진단 시 L27~52)에 하드코딩된 한국어(`<h1>내 주변</h1>` 등)를 i18n 라벨로 치환. 첫 화면이 선택 언어로 뜨게 한다.
- **`placeSeedNote`의 "현재 가게 정보는 영업용 시드입니다" 등 내부 영업용 문구를 사용자 화면에서 제거** (마케팅·디자인 공통 지적). 미검증 점포는 노출하지 않거나, "정보 확인 중" 수준의 사용자 친화 문구로 대체.
- 위치 라벨 버그: geolocation 성공 시 버튼 라벨이 `placeUi("nearby")`("내 주변")로 덮여 실제 동네명이 안 나온다(진단 시 app.js L1640 부근). 역지오코딩(카카오 로컬)으로 동(洞) 단위 지명 표시, 불가 시 "현재 위치"처럼 명시적 카피로.

### 완료 기준 (3장)
- 한국어 못 읽는 사용자가 앱을 켜면, **첫 화면부터 끝까지 선택 언어로만** 보인다.
- 언어를 바꾸면 발견·배우기·UI 라벨이 **모두 즉시** 그 언어로 바뀐다.
- 화면 어디에도 "영업용 시드" 같은 내부 문구가 없다.

## 4. P1 — 메인=발견 / 서브=배우기 재배치 + P2 데이터·메타

### 4-1. 위계 재배치
- 기본 진입 탭을 **발견(정보공유)** 으로 고정(현재 `appTab:"nearby"`는 발견이 맞으나, 학습앱 정체성의 카피·메타가 메인을 가린 상태).
- "배우기"는 보조 탭으로 명확히 강등(내비 순서·시각 위계). 기능 삭제 금지(서브 후크로 유지).
- 두 탭의 헤더 디자인 언어 통일: `placeHeader`(오렌지/그린)와 `hero`(남색/태극) 그라데이션을 한 시스템으로 수렴. 브랜드 액센트를 `--accent`(틸) 하나로, 발견은 보조색으로.

### 4-2. PLACE_COPY 7개 언어 확장 (P2)
- `data/places.js`의 `PLACE_COPY`, `PLACE_CATEGORY`/`NATIONALITY_LABELS` 등 정적 라벨을 **학습 데이터와 동일한 7개 언어(ko/en/th/vi/ja/zh/es)로 확장**.
- ⚠️ 번역 텍스트는 codex가 임의 생성하지 말 것. **번역이 필요한 신규 문자열 목록을 뽑아 기획(Claude)에 넘기면 다국어 텍스트를 작성/검수**해서 제공한다. (회사 규칙: 다국어 텍스트는 기획 담당, 원어민 검수 권장)

### 4-3. 메타·OG·manifest·privacy 피벗 반영 (P2)
현재 전부 구 제품(학습앱)에 고정돼 있어 공유·SEO·앱설치 미리보기가 "한국어 학습앱"으로 뜬다.
- `<title>`, `og:*`, `twitter:*` → 로컬 발견 메시지로 교체 (예: "내 동네 로컬 맛집·마트를 내 언어로 — wiiInfo")
- `manifest.webmanifest` `name`/`description`/`theme_color` 교체
- og-image 재제작 필요 → 이미지 시안은 기획/디자인에서 별도 제공 (codex는 교체 적용만)
- **`privacy.html`에 위치정보 사용·점포 데이터·(향후) 쿠폰·점주 정보 수집을 명시**. 위치 권한을 받는 앱이 privacy에 위치 언급이 없으면 스토어 심사·신뢰 양쪽에서 문제. 현재 privacy는 학습/로그인만 다룸.

## 5. P2 — 획득 루프 (사업 핵심, 제품에 장치를 심는다)

> **핵심 성장 전략 (형님 확정):**
> **영업 가서 업주 등록시킬 때 → 그 가게에 모국어 QR 포스터를 붙임 → 그 가게에 오는 외국인 손님이 곧 첫 사용자가 됨.**
> 즉 **영업 1회 = 업주(공급) + 그 가게 단골 손님(수요)** 을 동시에 확보한다. 별도 광고비 없이 점포 매대가 광고판이 된다. 양면시장에서 공급만 채우면 수요가 자동으로 오지 않으므로, 이 루프를 제품에 내장해 "시간이 복리로 쌓이게" 한다.

이 전략을 제품이 뒷받침하려면 다음 기능이 필요하다:
1. **점포별 딥링크/공유 URL** — 특정 점포 상세로 바로 진입하는 URL(예: `?place=<id>&lang=<th>`). QR이 이 링크를 가리킨다. 언어 파라미터로 진입 시 그 언어로 바로 뜨게(3장 언어 전역화와 연동).
2. **QR 코드 생성 동선** — 점포별 공유 링크의 QR을 만들 수 있는 경로(관리자/영업용). 인쇄용 포스터에 들어갈 형태. (포스터 디자인은 디자인 영역, 링크·QR 생성 로직은 codex)
3. **공유 버튼** — 점포 상세에 "공유하기"(Web Share API) — 사용자가 자국 커뮤니티(오픈채팅/페북)에 직접 퍼뜨리는 시딩 채널. 마케팅팀이 초기 획득 채널로 "국적별 커뮤니티 시딩 + 점포 QR"을 지목.

### 완료 기준 (5장)
- 점포별 고유 공유 URL이 존재하고, 그 URL로 들어오면 해당 점포가 **방문자 언어로** 바로 열린다.
- 영업 담당이 점포별 QR을 뽑을 수 있다.

## 6. P3 — 기술 정리 (출시 폴리싱)

근거: 개발·디자인 진단.

1. **초기 로딩 분리** — `multilingual-data.js`(105KB)가 첫 화면(발견)에 불필요한데 동기 블로킹 로드된다(index.html 진단 시 L254~256). `defer` 부여 또는 배우기 탭 진입 시점 지연로드로 전환 → 첫 진입 페이로드 100KB+ 절감.
2. **info-guide 캐시 정책 일원화** — app.js는 지연로드(L65~79)인데 sw.js `CORE_ASSETS`(L21)는 215KB를 선캐시해 정책이 모순. 지연로드 유지면 CORE_ASSETS에서 빼 runtime 캐시로, 오프라인 보장이 목적이면 의도를 주석 명시. 둘 중 하나로 통일.
3. **demo.html 서빙 제외** — 죽은 코드인데 `https://wiiinfo.web.app/demo.html`로 외부 노출 중. `firebase.json` ignore에 `"demo.html"` 추가(파일은 회사 규칙대로 보존, 서빙만 차단).
4. **로딩/에러 상태 추가** — 현재 빈 상태(emptyState)만 있고 스켈레톤/스피너/재시도 없음. 발견 리스트·오디오 재생·네트워크 실패에 시각 피드백 추가(느린 네트워크 사용자 다수).
5. **지도 폴백 정직화** — 카카오 키 없을 때 "가짜 격자 + 개발 안내문(키를 넣으면…)"이 사용자에게 노출됨. 정적 지도 이미지 또는 "목록으로 보기" 유도로 교체하고 개발용 문구는 노출 금지.
6. **다크모드 도입** — `:root` 토큰이 잘 잡혀 있어 저비용. `@media (prefers-color-scheme: dark)` 변형 추가.
7. **복사 버튼 리스너 누수 제거** — `openInfoDetail()`에서 `.copyAction`을 열 때마다 재바인딩 → 이벤트 위임으로 변경.
8. **(선택) 최소 린트 도입** — 빌드도구 없는 Vanilla JS의 최대 리스크(전역 오타·미정의 참조) 차단용 `package.json` + ESLint(no-undef/no-unused-vars). CI 없이 로컬 `npx eslint`만으로도 가치.

## 7. 작업 시 준수 규칙 (회사 공통)

- **기존 코드는 삭제하지 말고 주석 처리**로 보존(용량 크게 늘지 않는 범위). 교체 시 사유·날짜 주석.
- **다국어 텍스트·번역은 codex가 임의 생성 금지** → 필요한 문자열 목록을 뽑아 기획(Claude)에 요청, 검수된 텍스트를 받아 적용.
- 각 단계 후 **Playwright(Brave) 실브라우저로 검증**(기존 `tools/verify-*.js` 패턴). 콘솔 에러 0 확인.
- **자산 버전 동기화**: 변경 배포 시 `index.html`·`app.js` `ASSET_VERSION`·`sw.js` `SW_VERSION` 3곳 모두 bump(누락 시 캐시 불일치).
- 배포는 1개선 → 검증 → 커밋 → 배포 → push 워크플로우. 한 번에 하나씩.

4. **P1 메인/서브 재배치 + 헤더 통일** →
